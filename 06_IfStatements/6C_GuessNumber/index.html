If you collect the user's input as a string (i.e. without using parseInt() to turn it into an integer), does == work to compare the input to the stored secret number? Would === work? Which is best?

If we collect the user's input as a string, == does work when comparing the input to the stored secret number. However, in this case, === does not work. So in this instance == would be best since it is the only option that works, however, much like with the password example, I believe first converting the input to an integer and then comparing with === would be more preferential.

If you collect the user's input as an integer (i.e. by using parseInt() to turn it into an integer), can you use == to compare? ===? Which is better in this case?

If we collect the user's input as an integer, both == and === can be used successfully to compare. In this case, I don't think there is a clear choice as to which is best since the parseInt() serves as a catch all by attempting to convert all user input into an integer first. Thus, the only options would be an integer or an undefined variable.

What is the difference between == and ===?

The difference between == and === is that === compares both the value and type of the two entities being compared, whereas == will attempt to convert the second entity being compared, so they can be different data types as long as their value evaluates to be equal.

In this case, does it make more sense to leave the user's guess as a string, or to convert it to an integer? Why? What are the relative advantages / disadvantages of each approach (particularly in terms of the types of new features that might be able to be added)?

I think it makes more sense to conver the user's guess to an integer since the secret number is also stored as an integer. This then becomes a disadvantage, however, advantageous to the other approach, if the additional features will attempt to use string functions on either variable.
